# Scenario partition: demo vs reference

> Status: doctrine. Authoritative split between hand-curated golden
> fixtures and the generated reference cohort corpus that the
> fifth-kind loop uses as its measurement workload.

`dogfood/scenarios/` contains two functionally distinct populations of
scenarios. They serve different purposes, occupy disjoint ID ranges,
and live in disjoint subdirectories. **Do not confuse them.**

## `demo/` — Golden hand-seeded examples

- **ID range**: `[10000, 20000)`
- **Authorship**: Hand-curated. Each scenario is a deliberate, named
  example that documents expected pipeline behavior on a known input.
- **Stability contract**: Frozen. Changes require explicit doctrinal
  review because dozens of unit and integration tests pin against
  these exact files (currently `10001`, `10010`, `10011`).
- **Used by**: Unit tests, integration tests, fixture verification,
  CLI snapshots, regression pinning.
- **Lifecycle**: Lives forever. Effectively a golden file set.

## `reference/` — Generated reference cohort corpus

- **ID range**: `[20000, 30000)`
- **Authorship**: Generated by the cohort orchestrator
  (`lib/application/synthesis/cohort-generator.ts`). Twelve cohorts ×
  20 scenarios each = 240 scenarios in `[20000, 20240)`. The remaining
  `[20240, 30000)` is reserved for future cohort expansion.
- **Stability contract**: Deterministic regeneration from a master
  seed. Same seed → byte-identical output. Reseeding or reordering
  cohorts is a doctrinal change captured by the manifest's aggregate
  content hash.
- **Used by**: The fifth-kind loop's pipeline-measurement workload.
  Fitness scoring reads receipts produced by running this corpus
  through the pipeline; baselines and L4 metric deltas are computed
  against this corpus's output.
- **Lifecycle**: Regenerated only when the cohort definitions or
  master seed change. Otherwise read-only between runs — the substrate
  growth loop and the pipeline measurement loop both READ this corpus
  but never regenerate it as a side effect.

## Why the partition matters

The fifth-kind loop measures pipeline efficacy by running a **frozen
workload** through whatever pipeline code is on disk and diffing the
resulting metric tree against a prior baseline. If the workload churns
between commits, the gradient signal is meaningless — you'd be scoring
against a moving target.

The hand-seeded `demo/` scenarios cannot serve as that workload
because they are too small, too curated, and too coupled to test
pinning to ever change. The `reference/` cohort corpus exists
specifically to be **the immutable measurement workload** the
fifth-kind loop reads.

Conversely, the cohort corpus cannot serve as golden fixtures for
unit tests because it is regenerated whenever cohort definitions
change — pinning unit tests against generated content would create a
feedback loop where adding a cohort breaks unrelated tests.

The partition is enforced by:

- Disjoint ID ranges (`[10000, 20000)` vs `[20000, 30000)`)
- Disjoint directories (`demo/` vs `reference/`)
- A laws test (`tests/scenarios-id-partition.laws.spec.ts`) that
  asserts the partition holds for the reference cohort definitions
- The cohort generator's pre-flight overlap check, which refuses to
  write any file when cohort ID ranges collide with each other
