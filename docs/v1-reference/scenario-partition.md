# Scenario partition: demo vs reference

> Status: doctrine. Authoritative split between the legacy
> hand-curated golden fixtures (10000-series) and the generated
> reference cohort corpus (20000-series) that the self-improvement
> loop uses as its measurement workload.
>
> **Direction (2026-04-10 reframe).** The long-term goal is **one
> corpus of same-minded scenarios.** The 20000-series is the
> survivor. The 10000-series stays only because a handful of unit
> tests still pin against specific files there; those tests get
> migrated to their own fixture location under
> `tests/fixtures/` (or replaced by property-based generators) as
> part of the graduation-from-dogfood work (see
> `docs/canon-and-derivation.md` § 14.0). When the pins are gone,
> the 10000-series is deleted and the partition collapses. The
> longer-term end-state is that the dogfood loop does not exist at
> all because synthetic confidence has reached the point where a
> production ADO connection against a real SUT is the default
> operating mode.

`dogfood/scenarios/` contains two functionally distinct populations
of scenarios today. They serve different purposes, occupy disjoint
ID ranges, and live in disjoint subdirectories. **Do not confuse
them, and do not mix them** — the partition is a temporary
scaffolding that goes away during graduation.

## `demo/` — Legacy golden hand-seeded examples (10000-series)

- **ID range**: `[10000, 20000)`
- **Authorship**: Hand-curated. Each scenario is a deliberate, named
  example that documents expected pipeline behavior on a known input.
- **Stability contract**: Frozen DURING the graduation period.
  Changes require explicit doctrinal review because dozens of unit
  and integration tests pin against these exact files (currently
  `10001`, `10010`, `10011`).
- **Used by**: Unit tests, integration tests, fixture verification,
  CLI snapshots, regression pinning.
- **Lifecycle**: TRANSITIONAL. These scenarios exist because they
  underlie core unit tests that would break if removed. They are
  scheduled for migration to `tests/fixtures/` (or replacement by
  property-based generators) so the partition can collapse. Do
  NOT add new scenarios to this range — any new golden fixtures
  go under `tests/fixtures/` directly.

## `reference/` — The go-forward corpus (20000-series)

- **ID range**: `[20000, 30000)`
- **Authorship**: Generated by the cohort orchestrator
  (`lib/application/synthesis/cohort-generator.ts`). Twelve cohorts ×
  20 scenarios each = 240 scenarios in `[20000, 20240)`. The
  remaining `[20240, 30000)` is reserved for future cohort expansion.
- **Stability contract**: Deterministic regeneration from a master
  seed. Same seed → byte-identical output. Reseeding or reordering
  cohorts is a doctrinal change captured by the manifest's aggregate
  content hash. Per `docs/alignment-targets.md`, reseeding resets
  the M5 trajectory buffer for affected scenarios because
  "cohort-comparable" means "same scenario IDs."
- **Used by**: The self-improvement loop's pipeline-measurement
  workload. Fitness scoring reads receipts produced by running this
  corpus through the pipeline; baselines and L4 metric deltas
  (pipeline-efficacy tree + discovery-fitness tree) are computed
  against this corpus's output.
- **Lifecycle**: This is the **surviving corpus**. New workload
  expansions, new cohort definitions, new measurement workloads all
  land here. Regenerated only when the cohort definitions or master
  seed change. Otherwise read-only between runs.

## Why the partition matters

The fifth-kind loop measures pipeline efficacy by running a **frozen
workload** through whatever pipeline code is on disk and diffing the
resulting metric tree against a prior baseline. If the workload churns
between commits, the gradient signal is meaningless — you'd be scoring
against a moving target.

The hand-seeded `demo/` scenarios cannot serve as that workload
because they are too small, too curated, and too coupled to test
pinning to ever change. The `reference/` cohort corpus exists
specifically to be **the immutable measurement workload** the
fifth-kind loop reads.

Conversely, the cohort corpus cannot serve as golden fixtures for
unit tests because it is regenerated whenever cohort definitions
change — pinning unit tests against generated content would create a
feedback loop where adding a cohort breaks unrelated tests.

The partition is enforced by:

- Disjoint ID ranges (`[10000, 20000)` vs `[20000, 30000)`)
- Disjoint directories (`demo/` vs `reference/`)
- A laws test (`tests/scenarios-id-partition.laws.spec.ts`) that
  asserts the partition holds for the reference cohort definitions
- The cohort generator's pre-flight overlap check, which refuses to
  write any file when cohort ID ranges collide with each other
