# Convergence Roadmap: Closing the Gap Between Vision and Reality

> Status: Active — authoritative roadmap for the next phase of Tesseract development.

This document is a rigorous gap analysis and implementation roadmap grounded in the actual codebase as of 2026-04-03. It identifies the precise architectural seams where work is needed, explains the causal relationships between gaps, and prescribes implementation strategies that respect the existing invariants documented in `docs/master-architecture.md`, `docs/coding-notes.md`, and `docs/seams-and-invariants.md`.

The goal is not to enumerate features. The goal is to explain *why* the system converges at 4–9% cold-start hit rate instead of 40%+, and what structural changes — in what order — will move that number.

## 1. Diagnosis: Why Convergence Is Slow

### 1.1 The Observed Phenomenon

A cold-start diagnostic speedrun — 50 synthetic scenarios, 2 iterations, no prior knowledge — produces approximately 220 proposals in iteration 1 and 142 in iteration 2, with a knowledge hit rate that climbs from 0% to approximately 2–4%. A warm-start run with accumulated knowledge reaches 6–9%. The convergence FSM declares `max-iterations` termination, not `plateau` or true convergence.

The vision promises a system that "understands the DOM instead of repeatedly poking at it" and "remembers selector and state knowledge once." At 4–9% hit rate, 91–96% of scenario steps still fall through the entire resolution ladder to `needs-human` (rung 11). The system is generating proposals prolifically — 362 proposals activated across 2 cold-start iterations — but those proposals do not translate into resolution wins at rungs 3–6 on subsequent iterations at a rate that would produce perceptible convergence.

### 1.2 Root Cause Decomposition

The low convergence rate is not a single defect. It is the product of four interacting structural bottlenecks:

**Bottleneck 1: Widget family coverage is narrow.** The runtime has three widget handlers (`os-button.ts`, `os-input.ts`, `os-table.ts`) covering six action types (`click`, `fill`, `clear`, `get-value`). The demo harness itself only contains `<input>`, `<button>`, `<table>`, and `<a>` elements. Common form widgets — `<select>`, `<input type="checkbox">`, `<input type="date">`, tabs, modals, comboboxes, radio groups, accordions — are absent from both the harness and the widget handler registry. When a synthetic scenario references an action that maps to an uncovered widget type, the step cannot resolve regardless of how much knowledge accumulates. The ceiling is set by widget coverage, not by the learning loop.

**Bottleneck 2: Route knowledge does not exist.** The `routeSelectionPrecedenceLaw` in `lib/domain/resolution/precedence.ts:21-26` declares a rung for `route-knowledge`, but no `knowledge/routes/` directory exists and no route resolution implementation backs the declaration. The runtime navigates to screens using only canonical URL fallback. This means every scenario step that requires navigating to a screen in a specific state (e.g., `/policies?id=12345` for `policy-detail` with entity context, or `/policies?tab=claims` for a specific tab) falls through to degraded resolution. Route knowledge would allow the semantic dictionary and approved-knowledge rungs to resolve navigation steps that currently reach `needs-human`.

**Bottleneck 3: Proposal quality is minimal.** When a step falls to `needs-human`, the proposal generated by `proposalsForNeedsHuman()` in `lib/runtime/agent/proposals.ts:115-170` contains `{ screen, element, alias }` — sufficient to propose an element alias on a screen's hints file, but lacking richer semantic context. The proposal does not carry the action intent, the widget type, the DOM context that was available at resolution time, or the locator strategy that partially matched. When these minimal proposals activate into `knowledge/screens/*.hints.yaml`, the resulting hints are thin: they provide an alias mapping but not the locator ladder, default values, or widget affordance metadata that would let rungs 3–5 resolve the step deterministically on the next iteration. The activation loop closes — proposals become knowledge — but the knowledge is too sparse to shift resolution from rung 11 to rung 3.

**Bottleneck 4: The demo harness constrains the exposure surface.** The demo harness (`dogfood/fixtures/demo-harness/`) serves three screens (`policy-search`, `policy-detail`, `policy-amendment`) plus a React journey page. These screens contain approximately 15–20 unique interactable elements across basic widget types. Synthetic scenarios generated from seed templates exercise permutations of these elements, but the total vocabulary of distinct (screen, element, action) triples is small. The learning loop saturates quickly because there are few novel interactions to discover. The entropy harness (D1) — which would inject phrasing variants, widget swaps, and layout mutations — is only partially implemented (drift event tagging exists, but no variance profile injection).

### 1.3 The Interaction Effect

These bottlenecks compound. A step referencing a dropdown widget on a route-parameterized screen will fail at three levels simultaneously:

1. No route knowledge to navigate to the correct screen state (bottleneck 2)
2. No widget handler to execute the dropdown interaction (bottleneck 1)
3. The proposal generated at `needs-human` will be too sparse to teach the system how to handle dropdowns in the future (bottleneck 3)

Each bottleneck individually caps convergence. Together, they create a ceiling well below the 40%+ hit rate that would make the system feel intelligent rather than mechanical.

### 1.4 What Is Not a Bottleneck

Several aspects of the system are *not* limiting convergence:

- **The resolution ladder is complete.** All 11 rungs are implemented and wired in `lib/runtime/agent/index.ts:188-227`. The fallthrough DAG (`explicit → control → approved-knowledge → patterns → evidence → semantic-dict → overlay → translation → live-dom → agent → needs-human`) executes correctly.
- **The convergence FSM is sound.** The state machine in `lib/domain/projection/convergence-fsm.ts` correctly transitions through `exploring → narrowing → plateau → converged` with proper absorbing-state semantics.
- **The activation loop is closed.** As of the recent act-loop feedback arc change, proposals approved by the decider are written to disk via `tryActivateProposal()`. The bookkeeping-only gap is fixed.
- **The semantic dictionary works.** Entries accrue with initial confidence 0.5, boost by 0.12 per successful reuse, and promote to approved knowledge at confidence 0.8 after 3+ reuses. The similarity threshold (Jaccard + TF-IDF at 0.45) and combined score threshold (0.35) are reasonable.
- **Auto-approval gates are functional.** The 6-gate policy (`enabled → artifact-type → confidence → evidence → trust-policy → per-run-limit`) correctly filters proposals.
- **Rung 8 (structured translation) is deterministic.** Despite the name, `rung8-llm-dom.ts` performs pure signal extraction over ARIA snapshots — no LLM call. It extracts `aria-label-match`, `role-match`, `text-content-match` signals and computes confidence from signal count. This is a strength, not a limitation: it means diagnostic mode exercises the same code path as production.
- **The MCP server is comprehensive.** 27 tools across observation, decision, browser, analysis, and lifecycle categories. The agent integration surface is mature.

## 2. The Convergence Bottleneck Graph

The four bottlenecks form a directed acyclic graph of causal dependencies. Understanding this graph is essential for correct sequencing — addressing a downstream bottleneck before its upstream dependency yields no convergence improvement.

```
                    ┌─────────────────────┐
                    │  Demo Harness Scope  │
                    │   (Bottleneck 4)     │
                    └──────────┬──────────┘
                               │ constrains
                    ┌──────────▼──────────┐
                    │  Widget Coverage     │
                    │   (Bottleneck 1)     │
                    └──────────┬──────────┘
                               │ caps
              ┌────────────────┼────────────────┐
              │                │                │
   ┌──────────▼─────┐  ┌──────▼──────┐  ┌──────▼──────────┐
   │ Route Knowledge │  │  Proposal   │  │  Entropy        │
   │  (Bottleneck 2) │  │  Quality    │  │  Harness        │
   │                 │  │ (Bottleneck │  │  (Amplifier)     │
   │                 │  │     3)      │  │                  │
   └────────┬────────┘  └──────┬──────┘  └──────┬──────────┘
            │                  │                │
            └──────────────────┼────────────────┘
                               │
                    ┌──────────▼──────────┐
                    │  Convergence Rate   │
                    │  (Observable)       │
                    └─────────────────────┘
```

### 2.1 Reading the Graph

**Widget coverage (B1)** is the load-bearing constraint. It determines the maximum set of (action, widget-type) pairs the runtime can execute. Every other bottleneck operates within this ceiling.

**Route knowledge (B2)** and **proposal quality (B3)** are independent peers that both feed convergence. Route knowledge determines whether the runtime can reach the correct screen state. Proposal quality determines whether the learning loop produces knowledge rich enough to shift resolution from rung 11 to rungs 3–6.

**The demo harness (B4)** constrains widget coverage from above: you cannot add a dropdown handler if the harness has no dropdowns to test against.

**Structured entropy (amplifier)** is not a bottleneck in the strict sense — it is a multiplier on the learning rate. With the same widget coverage and proposal quality, higher entropy (more phrasing variants, more state permutations, more layout mutations) produces more diverse proposals per iteration, accelerating convergence. Without it, the loop saturates on repeated identical interactions.

### 2.2 Implication for Sequencing

The graph dictates a clear priority order:

1. **Expand the demo harness** with missing widget types (enables everything else)
2. **Add widget handlers** for the new widget types (raises the ceiling)
3. **Enrich proposal content** so activated knowledge is dense enough to resolve at rungs 3–6 (improves yield per iteration)
4. **Implement route knowledge** so navigation steps resolve deterministically (opens a new category of resolvable steps)
5. **Build the entropy harness** to accelerate convergence within the new ceiling (multiplier on all of the above)

This is the order in which the workstreams below are presented. Workstreams E (test suite) and F (operational surface) are orthogonal — they can proceed in parallel with any of the above.

> **Revision (2026-04-03):** Workstream A has been revised. The original proposal to hand-author widget handlers per type was identified as an alias treadmill — the same antipattern that A1 (ADR collapse) eliminated for step text resolution. The revised approach derives widget affordances deterministically from ARIA role signatures. See the execution backlog at `docs/convergence-backlog.md` for the detailed implementation plan.

## 3. Workstream A: Widget Family Coverage

### 3.1 Current State

The widget handler registry (`lib/runtime/widgets/index.ts`) contains three entries:

| Widget | File | Actions | Playwright Operations |
|--------|------|---------|----------------------|
| `os-button` | `os-button.ts` | `click`, `get-value` | `locator.click()`, `locator.innerText()` |
| `os-input` | `os-input.ts` | `fill`, `clear`, `get-value` | `locator.fill(value)`, `locator.fill('')`, `locator.inputValue()` |
| `os-table` | `os-table.ts` | `get-value` | (stub) |

The `WidgetAction` union type in `lib/domain/types/widgets.ts:4` is restricted to `'click' | 'fill' | 'clear' | 'get-value'`. The `WidgetCapabilityContract` interface provides the extensibility surface: each widget declares `supportedActions`, `requiredPreconditions`, and `sideEffects` per action.

The demo harness contains only `<input>`, `<button>`, `<table>`, and `<a>` elements — no dropdowns, checkboxes, date pickers, modals, or tabs.

### 3.2 Target State

The widget registry should cover the standard form vocabulary that enterprise insurance applications use. Based on OutSystems and similar low-code platform patterns, the minimum viable coverage is:

| Widget | Actions | Playwright Operations | Priority |
|--------|---------|----------------------|----------|
| `os-select` | `select`, `get-value`, `get-options` | `locator.selectOption()`, `locator.inputValue()`, `locator.locator('option').allInnerTexts()` | P0 |
| `os-checkbox` | `check`, `uncheck`, `get-value` | `locator.check()`, `locator.uncheck()`, `locator.isChecked()` | P0 |
| `os-radio` | `select`, `get-value` | `locator.check()`, `locator.isChecked()` | P0 |
| `os-datepicker` | `fill`, `clear`, `get-value` | `locator.fill(value)` with date formatting | P1 |
| `os-combobox` | `fill`, `select`, `get-value`, `get-options` | `locator.fill()` + `locator.locator('[role=option]')` | P1 |
| `os-tabs` | `select-tab`, `get-active-tab` | `locator.getByRole('tab', { name }).click()`, attribute query | P1 |
| `os-modal` | `open`, `close`, `get-value` | Dialog handling, `locator.getByRole('dialog')` | P2 |
| `os-accordion` | `expand`, `collapse`, `get-value` | Click on header, check `aria-expanded` | P2 |
| `os-link` | `click`, `get-value` | `locator.click()`, `locator.getAttribute('href')` | P0 |

### 3.3 Implementation Strategy

**Step 1: Extend the domain type.** The `WidgetAction` union in `lib/domain/types/widgets.ts` must grow to include `'select' | 'check' | 'uncheck' | 'select-tab' | 'get-options' | 'get-active-tab' | 'open' | 'close' | 'expand' | 'collapse'`. This is a closed-set change — the union is exhaustively matched in the widget capability derivation and resolution ladder.

**Step 2: Add widget handler files.** Each new widget gets a file in `lib/runtime/widgets/` following the existing pattern:

```typescript
// lib/runtime/widgets/os-select.ts
import type { Locator } from '@playwright/test';

export const osSelectHandlers: Record<
  'select' | 'get-value' | 'get-options',
  (locator: Locator, value?: string) => Promise<unknown>
> = {
  async select(locator, value = '') {
    await locator.selectOption(value);
  },
  async 'get-value'(locator) {
    return locator.inputValue();
  },
  async 'get-options'(locator) {
    return locator.locator('option').allInnerTexts();
  },
};
```

**Step 3: Register in the handler registry.** Add each new widget to `widgetActionHandlers` in `lib/runtime/widgets/index.ts`.

**Step 4: Add widget capability contracts.** For each new widget, define a `WidgetCapabilityContract` that declares supported actions, preconditions, and side effects. These contracts participate in the resolution ladder at rung 3 (`approved-screen-knowledge`) when hints reference a widget affordance.

**Step 5: Expand the demo harness.** Add corresponding HTML elements to the demo harness screens. For example, add a `<select>` for policy status filter, a `<input type="checkbox">` for agreement confirmation, and tabs for the policy detail view.

**Step 6: Update seed templates.** The synthetic scenario generator must reference the new widget types in its phrasing templates so that speedrun scenarios exercise the new handlers.

### 3.4 Invariants to Preserve

- Widget handlers remain thin Playwright-only adapters. No domain logic, no orchestration.
- The `WidgetCapabilityContract` is the contract surface — handlers implement it, resolution queries it.
- New actions must be added to the closed `WidgetAction` union and handled exhaustively in any `switch` or fold that consumes it.
- The handler registry is the sole dispatch surface. No ad hoc handler lookup elsewhere in the runtime.

### 3.5 Expected Convergence Impact

Adding P0 widgets (`os-select`, `os-checkbox`, `os-radio`, `os-link`) with corresponding harness elements and seed templates should raise the widget-coverable action vocabulary from 4 actions to 10+. This directly raises the ceiling on the percentage of synthetic scenario steps that *can* resolve, independent of knowledge quality. Conservative estimate: the resolvable-step ceiling moves from ~40% to ~70% of generated step vocabulary.

## 4. Workstream B: Route Knowledge and Navigation

### 4.1 Current State

Route knowledge is more developed than a surface audit suggests. Three layers exist:

**Domain layer (implemented).** `lib/domain/knowledge/route-knowledge.ts` provides pure functions for URL segment analysis, route pattern derivation, parameter extraction, and route variant ranking via `rankRouteVariants()`. The type system in `lib/domain/types/route-knowledge.ts` defines `ObservedRoute`, `RoutePattern`, `RouteVariantKnowledge`, `RankedRouteVariant`, and `RouteKnowledgeProposal`. This is a complete domain model.

**Runtime layer (wired).** `lib/runtime/scenario.ts:215-223` already resolves route selection through the `routeSelectionPrecedenceLaw` precedence ladder (`explicit-url → runbook-binding → route-knowledge → screen-default`). The `route-knowledge` rung is wired and will resolve if route variant data is present in the workspace session.

**Application layer (partially wired).** `lib/application/workspace-session.ts:31-64` loads route knowledge from harvest manifests into a `routeKnowledge: Map<ScreenId, RouteVariant[]>` structure. `lib/application/discovery-proposal-bridge.ts:182-187` generates `route-knowledge` proposals during discovery runs.

**What is missing.** No `knowledge/routes/` directory exists. The discovery pipeline (`npm run discover`) produces route observations, but these are not persisted as reviewed knowledge artifacts in the format that the workspace session loader expects. The harvest manifests populate `routeKnowledge` transiently from `.tesseract/interface/` artifacts, but there is no canonical persistence path where an operator can review and approve route knowledge the way they review `knowledge/screens/*.hints.yaml`.

### 4.2 The Gap: Discovery-to-Knowledge Persistence

The route knowledge pipeline has a persistence gap at one specific seam:

```
Discovery run                Harvest manifest              Workspace session
(npm run discover)    →    (.tesseract/interface/)    →    (transient Map)
     ✅                          ✅                           ✅

                                    ↓ should persist to

                        knowledge/routes/{app}.routes.yaml
                                    ❌ MISSING
```

The domain model, the ranking function, the runtime precedence wiring, and the workspace session loader all exist. The missing piece is a persistence adapter that writes route observations to `knowledge/routes/` in a schema that:

1. Maps URL patterns to screen IDs
2. Records query parameters and their UX effects
3. Carries provenance back to the discovery run that observed the route
4. Participates in the same governance model as other knowledge artifacts

### 4.3 Implementation Strategy

**Step 1: Define the route knowledge schema.** Create `knowledge/routes/{app}.routes.yaml` with a schema derived from the existing `RouteVariantKnowledge` type:

```yaml
kind: route-knowledge
version: 1
app: demo
routes:
  - screen: policy-search
    pattern: /search
    variants:
      - name: default
        url: /search
        params: {}
      - name: results-with-policy
        url: /search?policyNumber={policyId}
        params:
          policyNumber: { kind: entity-id, source: dataset }
  - screen: policy-detail
    pattern: /policies/{policyId}
    variants:
      - name: with-policy
        url: /policies/{policyId}
        params:
          policyId: { kind: entity-id, source: dataset }
```

**Step 2: Write the persistence adapter.** Add a function in `lib/infrastructure/knowledge/` that writes `RouteKnowledgeProposal` artifacts to `knowledge/routes/`. This follows the same pattern as the hints writer (`lib/infrastructure/knowledge/hints-writer.ts`).

**Step 3: Wire discovery proposals to persistence.** The `discovery-proposal-bridge.ts` already generates `route-knowledge` proposals. Add a post-discovery step that persists accepted proposals to `knowledge/routes/`.

**Step 4: Update workspace session loading.** Extend `lib/application/workspace-session.ts` to load from `knowledge/routes/` in addition to harvest manifests. Route knowledge from authored YAML should take precedence over transient harvest data (it is reviewed and approved).

**Step 5: Seed route knowledge for the demo harness.** Author `dogfood/knowledge/routes/demo.routes.yaml` with the three demo screens and their known URL patterns. This immediately makes route-knowledge rung 3 resolvable for all demo navigation steps.

### 4.4 Invariants to Preserve

- Route knowledge YAML is canonical and reviewable — it follows the same governance model as hints and patterns.
- The `routeSelectionPrecedenceLaw` order must not change. `explicit-url` > `runbook-binding` > `route-knowledge` > `screen-default`.
- Route knowledge does not bypass the dataset binding for entity IDs — it references dataset parameters, it does not hard-code values.
- Discovery proposals remain proposals — they do not auto-persist without trust policy evaluation.

### 4.5 Expected Convergence Impact

Navigation steps constitute roughly 10–15% of synthetic scenario step vocabulary (navigate to screen, open URL, go to page). With route knowledge, these steps resolve at rung 3 (`route-knowledge`) instead of falling through to `needs-human`. Combined with widget coverage expansion, this raises the resolvable ceiling by an additional 10–15 percentage points.

## 5. Workstream C: Proposal Quality and Activation Yield

### 5.1 Current State

When a step falls to `needs-human` (rung 11), `proposalsForNeedsHuman()` in `lib/runtime/agent/proposals.ts:115-170` generates a `ResolutionProposalDraft` with a patch of the form:

```typescript
{
  screen: 'policy-search',
  element: 'policyNumberInput',
  alias: 'Enter the policy number',
}
```

When this proposal is activated by `tryActivateProposal()`, the patch is applied by `applyHintsPatch()` in `lib/application/proposal-patches.ts:29-69`. The result is that the element's `aliases` array in the screen's hints YAML gains one new entry:

```yaml
elements:
  policyNumberInput:
    aliases:
      - "Enter the policy number"
    acquired:
      certification: uncertified
      activatedAt: "2026-04-03T..."
```

This is the *entire* knowledge contribution of one proposal. The element gains an alias — a text-matching anchor that lets rung 3 (`approved-screen-knowledge`) recognize the same phrasing in future iterations.

### 5.2 Why This Is Insufficient

The alias is necessary but not sufficient for robust resolution. Consider what a fully-authored hints entry looks like:

```yaml
elements:
  policyNumberInput:
    aliases:
      - "Enter the policy number"
      - "Type policy number"
      - "Policy number field"
    locatorLadder:
      - kind: test-id
        value: "policy-number-input"
      - kind: role-name
        value: "textbox"
        name: "Policy Number"
      - kind: css
        value: "#policy-number-input"
    defaultValue: "POL-001"
    widgetAffordance: os-input
    parameterCues:
      - dataset: policies
        field: policyNumber
```

The proposal contributes *one alias*. It does not contribute the locator ladder, the widget affordance, the default value, or the parameter cues. Without these, the resolution ladder can match the alias at rung 3 but cannot *execute* the step — it lacks the locator strategy to find the element and the widget type to interact with it.

The result: the step resolves at rung 3 for intent matching but still requires runtime DOM exploration (rung 9) for actual execution. The convergence metric counts this as a "hit" at rung 3, but the execution cost is not reduced — the runtime still discovers the locator at execution time.

### 5.3 The Enrichment Opportunity

At the moment a proposal is generated in `proposalsForNeedsHuman()`, the resolution pipeline has already accumulated significant context that is discarded:

1. **DOM signals from rung 8** (`rung8-llm-dom.ts`): ARIA snapshot analysis produces `aria-label-match`, `role-match`, `text-content-match` signals with associated selectors.
2. **Candidate lattice entries**: The `StepTaskElementCandidate` carries `element`, `confidence`, and `evidence` including `locatorStrategy`.
3. **Widget type inference**: The action text ("Enter", "Click", "Select") implies a widget type that could be recorded as `widgetAffordance`.
4. **Evidence from prior runs**: The evidence drafts generated alongside proposals carry `locatorStrategy` and `score` fields.

None of this context is captured in the proposal patch. It is logged in evidence drafts (`.tesseract/evidence/`) but not fed back into the knowledge activation loop.

### 5.4 Implementation Strategy

**Step 1: Enrich the proposal patch schema.** Extend `ResolutionProposalDraft.patch` to include optional fields that the activation loop can consume:

```typescript
patch: {
  screen: string;
  element: string;
  alias: string;
  // New optional enrichment fields:
  locatorHints?: Array<{ kind: string; value: string }>;
  widgetAffordance?: string;
  defaultValue?: string;
  inferredAction?: string;
}
```

**Step 2: Populate enrichment fields in `proposalsForNeedsHuman()`.** When a candidate element is available, extract its locator strategy and DOM signals:

```typescript
const locatorHints = element?.evidence?.locatorStrategy
  ? [{ kind: element.evidence.locatorStrategy, value: element.evidence.selector }]
  : rung8Signals.filter(s => s.selector)
    .map(s => ({ kind: s.signalType, value: s.selector }));

const widgetAffordance = inferWidgetFromAction(task.actionText);
```

**Step 3: Extend `applyHintsPatch()` to consume enrichment fields.** When the patch includes `locatorHints`, merge them into the element's `locatorLadder` array (deduplicating by kind). When it includes `widgetAffordance`, set the element's affordance field if not already present.

```typescript
const updatedEntry: Record<string, unknown> = {
  ...elementEntry,
  aliases: withAlias(existingAliases, alias),
  ...(patch.locatorHints && !elementEntry.locatorLadder
    ? { locatorLadder: patch.locatorHints }
    : {}),
  ...(patch.widgetAffordance && !elementEntry.widgetAffordance
    ? { widgetAffordance: patch.widgetAffordance }
    : {}),
  acquired: { ... },
};
```

**Step 4: Enrich evidence-based proposals.** When a step resolves at rung 8 or 9 (structured translation or live-DOM) rather than falling to `needs-human`, generate a *supplementary* proposal that captures the successful resolution's locator and widget type. These proposals have lower priority than `needs-human` proposals but contribute locator knowledge for elements that are already aliased.

### 5.5 Invariants to Preserve

- The `{ screen, element, alias }` triple remains the minimum valid patch. Enrichment fields are additive, never required.
- Enrichment fields use `??` (set-if-absent) semantics — they never overwrite hand-authored knowledge. An operator-authored locator ladder always takes precedence over a proposal-inferred one.
- The `applyHintsPatch()` function remains the sole knowledge-write path. No direct YAML manipulation in the proposal pipeline.
- Evidence drafts remain separate from proposals — proposals are knowledge patches, evidence is runtime observation. The enrichment bridge extracts from evidence at proposal-generation time, not at activation time.

### 5.6 Expected Convergence Impact

With enriched proposals, a single `needs-human` occurrence for a novel element produces a hints entry that includes not just an alias but also a locator hint and widget type. On the next iteration, rung 3 can resolve the step *and* the runtime can execute it without falling back to DOM exploration. This converts partial hits (alias match, execution fallback) into full hits (alias match, locator match, widget match).

Conservative estimate: enriched proposals should convert 30–50% of the proposals that currently produce alias-only hints into proposals that produce full-resolution hints. Combined with the widget coverage ceiling lift from Workstream A, this translates to a 15–25 percentage point improvement in effective hit rate (steps that resolve *and execute* at rung 3–6).

## 6. Workstream D: Structured Entropy and Variance Profiles

### 6.1 Current State

Drift events are defined and tracked. `BenchmarkDriftEvent` in the domain types supports `kind: 'locator-degradation' | 'widget-swap'`, and benchmark infrastructure references `driftEventIds` for correlation. However, the system that *injects* variance — the structured entropy harness described in BACKLOG.md item D1 — does not exist as a pluggable system.

Variance currently enters the system through two implicit channels:

1. **Scenario generation seed templates.** The `scripts/speedrun.ts` pipeline generates synthetic scenarios by permuting seed templates with action phrases and field names. This produces phrasing variance (the same intent expressed differently) but not structural or execution variance.
2. **Per-iteration proposal accumulation.** Each iteration's proposals carry slightly different alias phrasings based on the specific step text that triggered `needs-human`. Over multiple iterations, the hints files accumulate multiple aliases per element, providing implicit phrasing coverage.

Neither channel provides the controlled, measurable entropy that would accelerate knowledge hardening. The system encounters the same DOM structure, the same widget types, and the same navigation paths every iteration. Only the *phrasing* of the intent varies.

### 6.2 The Three Variance Dimensions

The backlog identifies three variance dimensions. Each targets a different layer of the resolution pipeline:

**Input variance** (what the system reads from scenarios):
- Synonym injection: "Click the Search button" vs "Press Search" vs "Hit the search btn"
- Passive voice: "The policy number should be entered" vs "Enter the policy number"
- Abbreviated forms: "Enter pol. no." vs "Enter policy number"
- Quantifier variants: "Enter a policy number" vs "Enter the policy number" vs "Enter policy number 12345"

Input variance exercises rungs 3–6 (alias matching, semantic dictionary, overlay resolution). It is the cheapest form of entropy — it requires only template mutations, no DOM changes.

**Structural variance** (what the DOM looks like):
- Salted `aria-label` changes: "Search" → "Search Policies" → "Find Policy"
- Widget swaps: `<select>` → `<div role="combobox">` for the same logical element
- Layout mutations: column reorder in result tables, field repositioning in forms
- Dynamic content: loading states, empty states, error states, populated states

Structural variance exercises rungs 8–10 (structured translation, live DOM, agent fallback). It is the most valuable form of entropy because it tests the system's ability to recover from DOM drift — the core promise of interface intelligence.

**Execution variance** (how the system runs):
- Interpreter mode toggles: `diagnostic` vs `playwright` for the same scenario
- Confidence threshold sweeps: strict trust policy vs permissive
- Rung restriction: force fallback to lower rungs by disabling higher ones
- Budget variation: 3 iterations vs 10 iterations on the same seed

Execution variance tests the system's robustness across operating configurations. It is measured through scorecard comparisons across variance profiles.

### 6.3 Implementation Strategy

**Step 1: Define variance profiles as declarative YAML artifacts.**

```yaml
# dogfood/variance/phrasing-diversity.variance.yaml
kind: variance-profile
version: 1
name: phrasing-diversity
dimensions:
  input:
    synonymInjection:
      enabled: true
      substitutionRate: 0.3  # 30% of action phrases get synonym variants
      sources:
        - knowledge/patterns/core.patterns.yaml  # reuse existing pattern aliases
    passiveVoice:
      enabled: true
      rate: 0.15
    abbreviation:
      enabled: true
      rate: 0.10
  structural:
    ariaLabelSalt:
      enabled: false  # requires demo harness mutation support
    widgetSwap:
      enabled: false
  execution:
    interpreterModes: [diagnostic, playwright]
    confidenceThresholds: [0.6, 0.8, 0.95]
```

**Step 2: Build the entropy injector.** A pure function in `lib/domain/` that takes a scenario and a variance profile and produces N mutated scenario variants:

```typescript
function applyVarianceProfile(
  scenario: ScenarioIR,
  profile: VarianceProfile,
  rng: SeededRandom,
): readonly ScenarioIR[];
```

The function is deterministic given a seed, so entropy is replayable and measurable.

**Step 3: Wire into the speedrun pipeline.** The `iteratePhase` in `lib/application/speedrun.ts` should accept an optional variance profile. When present, each scenario is expanded into N variants before compilation, multiplying the exposure surface.

**Step 4: Tag variance-injected scenarios.** Each mutated scenario carries a `varianceTag` in its metadata so that scorecard metrics can partition results by variance dimension. This is how you measure "does phrasing diversity actually improve alias coverage?" vs "does structural variance improve locator resilience?"

**Step 5: Implement structural variance via demo harness mutation.** This is the hardest step and can be deferred. It requires the demo harness to accept a mutation manifest that salts accessible names, swaps widget implementations, or reorders layouts. The harness server (`server.cjs`) would read the manifest at startup and apply mutations to the HTML templates.

### 6.4 Invariants to Preserve

- Variance profiles are declarative artifacts, not code changes.
- The entropy injector is a pure function — deterministic given a seed.
- Mutated scenarios carry provenance back to their source scenario and variance profile.
- Scorecard metrics partition by variance tag — aggregate metrics remain comparable across runs.
- Structural variance mutations are applied to the demo harness, not to knowledge artifacts. Knowledge remains canonical.

### 6.5 Expected Convergence Impact

Input variance (Step 1–4) is the highest-leverage intervention. By exposing the system to 3x–5x more phrasing variants per element, the alias accumulation rate per iteration increases proportionally. The semantic dictionary benefits especially — more diverse phrasings produce more dictionary entries, which raises the rung-6 hit rate.

Conservative estimate: input variance alone should improve per-iteration learning rate by 2x–3x, reducing the number of iterations needed to reach a given hit rate target. With 5 iterations instead of 2, cold-start hit rate should reach 15–25% (up from 4–9%).

## 7. Workstream E: Test Suite Rehabilitation

### 7.1 Current State

The test runner is Playwright Test (`@playwright/test`), invoked via `node ./node_modules/@playwright/test/cli.js test`. However, 11 spec files import test primitives from `vitest`:

```typescript
import { describe, it, expect } from 'vitest';
```

This causes immediate `Cannot find module 'vitest'` errors. The test suite cannot execute.

The tests themselves — law-style property tests for determinism, precedence, normalization, and round-trips — are structurally sound. The issue is purely an import mismatch: the files were authored expecting `vitest` as the runner, but the `package.json` `test` script invokes Playwright Test.

### 7.2 Implementation Strategy

There are two correct resolutions. The choice depends on whether the project intends to use Playwright Test or Vitest as the unit test runner.

**Option A: Migrate to Vitest (recommended).** The law-style tests are unit tests — they test pure domain functions with no browser interaction. Vitest is a better fit because it provides faster startup, native ESM support, and the `describe`/`it`/`expect` API the tests already use. Playwright Test should remain the runner for integration tests that require a browser.

1. Add `vitest` as a dev dependency.
2. Create `vitest.config.ts` that includes `tests/**/*.laws.spec.ts` and `tests/**/*.spec.ts` (excluding `tests/playwright-*.spec.ts`).
3. Update `package.json` `test` script to run `vitest run` for unit tests and `playwright test` for integration tests.
4. No changes needed to the spec files themselves — they already use the correct imports.

**Option B: Rewrite imports to Playwright Test.** Replace `import { describe, it, expect } from 'vitest'` with `import { test, expect } from '@playwright/test'` and refactor `describe`/`it` blocks to use Playwright's `test.describe`/`test` API. This is more work and produces worse ergonomics for pure function tests.

### 7.3 Invariants to Preserve

- Law-style tests must remain pure — no browser, no filesystem, no network.
- Integration tests must remain in Playwright Test — they exercise the real runtime against the demo harness.
- The `npm test` gate must run both kinds and fail on any regression.
- Test file naming conventions must remain stable: `*.laws.spec.ts` for property/law tests, `*.spec.ts` for behavioral tests, `playwright-*.spec.ts` for integration tests.

### 8.2 The Operator Cockpit (E1)

The vision describes an "operator cockpit" that turns inbox, workflow, trace, review, graph, and scorecards into one coherent surface. The most practical path to this is not a custom web UI — it is a structured MCP tool that synthesizes the existing projections into a single briefing:

**`get_operator_briefing` tool.** A new MCP tool that composes outputs from multiple existing tools:

```typescript
{
  phase: 'iterate',        // from get_loop_status
  iteration: 3,
  fitness: { ... },        // from get_fitness_metrics
  convergence: { ... },    // from get_convergence_proof
  topProposals: [...],     // from list_proposals (top 5 by priority)
  topHotspots: [...],      // from get_queue_items (top 5 investigate-hotspot)
  actionRequired: [...],   // from get_learning_summary
  recentDecisions: [...],  // from workbench completions
}
```

This is a composition — not a new data source. It reads the same artifacts the individual tools read, but synthesizes them into the briefing an operator needs to decide whether to intervene, adjust parameters, or let the loop continue.

### 8.3 CI Webhook Integration (F1)

The OutSystems Lifetime API integration is the most deployment-specific item in the backlog. It requires:

1. A webhook handler that receives module-publish events
2. A trigger that invokes `ci-batch` profile execution
3. Structured reports written to a known output path

This is pure infrastructure — no domain model changes, no compiler changes. It can be implemented as a standalone script in `scripts/ci-trigger.ts` that:

- Parses the Lifetime webhook payload
- Maps published modules to affected screens (using route knowledge when available)
- Invokes the pipeline in `ci-batch` mode for affected scenarios
- Writes structured reports to `.tesseract/runs/ci/`

The key invariant: `ci-batch` mode never auto-approves. Proposals accumulate for later operator review.

### 8.4 Confidence Decay (B3)

Confidence overlays at `.tesseract/confidence/index.json` currently have no decay mechanism. An overlay that was accurate six months ago may be stale if the application has changed. The backlog calls for configurable time-based decay.

Implementation: add a `decayPolicy` field to the trust policy configuration:

```yaml
# .tesseract/policy/trust-policy.yaml
decayPolicy:
  enabled: false        # off by default, per backlog spec
  halfLifeDays: 90      # confidence halves every 90 days
  minimumConfidence: 0.3 # floor below which the overlay is evicted
```

The decay calculation is a pure function over `(confidence, lastVerifiedAt, now, halfLifeDays)` — it belongs in `lib/domain/governance/trust-policy.ts` alongside the existing `evaluateAutoApproval()`. The application layer applies decay during catalog loading, before the resolution ladder consults overlays.

## 10. Measurement Framework

### 10.1 Primary Metric: Cold-Start Hit Rate at Iteration N

The single most important number is the knowledge hit rate after N iterations of a cold-start speedrun. This measures the system's ability to learn from scratch — the purest test of the learning loop.

| Milestone | Hit Rate | Iteration Budget | Interpretation |
|-----------|----------|-----------------|----------------|
| Current | 4–9% | 2–3 | System generates proposals but knowledge is too sparse to resolve |
| Post-Phase 1 | 10–15% | 3 | Widget coverage + route knowledge raise the ceiling |
| Post-Phase 2 | 20–35% | 5 | Enriched proposals + entropy accelerate learning |
| Vision target | 40%+ | 10 | System "understands the DOM" — majority of steps resolve from knowledge |

### 10.2 Secondary Metrics

| Metric | Source | Interpretation |
|--------|--------|----------------|
| Resolution by rung distribution | `get_fitness_metrics → resolutionByRung` | Shift from rung 11 toward rungs 3–6 indicates knowledge maturation |
| Proposal yield | `proposals activated / proposals generated` | Measures activation pipeline health |
| Effective hit rate | Steps resolved AND executed at rung ≤ 6 / total steps | Excludes alias-only matches that still require DOM fallback |
| Semantic dictionary promotion rate | Entries promoted to approved knowledge / total entries | Measures flywheel velocity |
| Convergence velocity | Iterations to reach plateau / total iterations | Lower is better; measures learning efficiency |
| Widget coverage ratio | Resolvable action types / total action types in scenario vocabulary | Measures ceiling set by widget handlers |

### 10.3 Measurement Protocol

Every workstream change should be validated with the same protocol:

```bash
# 1. Baseline: cold-start speedrun before the change
npx tsx scripts/convergence-proof.ts --trials 2 --count 50 --max-iterations 5 --mode diagnostic

# 2. Apply the change

# 3. Re-measure with identical parameters
npx tsx scripts/convergence-proof.ts --trials 2 --count 50 --max-iterations 5 --mode diagnostic

# 4. Compare scorecards
npx tsx scripts/mcp-call.ts get_fitness_metrics
npx tsx scripts/mcp-call.ts get_convergence_proof
```

The convergence proof script runs multiple trials to account for variance in scenario generation. The scorecard's Pareto frontier comparison and beat-the-mark gate detect regressions automatically.

### 10.4 The Convergence Proof as Acceptance Gate

The `get_convergence_proof` MCP tool already returns `{ converges: boolean, confidence, learningContribution }`. The acceptance gate for each workstream is:

1. `converges: true` — the system still converges (no regressions)
2. `learningContribution` increases — each iteration contributes more knowledge
3. Hit rate trajectory is monotonically non-decreasing — later iterations are at least as good as earlier ones
4. The specific bottleneck addressed by the workstream shows measurable improvement in its diagnostic metric

If a change passes the convergence proof but does not move the primary metric, the change is infrastructure (valuable but not convergence-impacting). If a change moves the primary metric but breaks the convergence proof, the change introduced a regression and must be revised.
