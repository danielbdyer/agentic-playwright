# VS Code and Copilot Ecosystem Survey

_As of March 10, 2026._

## Summary

- The current VS Code and Copilot surface is split across three different classes of capability:
  - repo-local customization surfaces: instructions, prompt files, Agent Skills, custom agents
  - extension-author APIs: Chat Participant API, Language Model API, Language Model Tool API, Task Provider API
  - runtime connectivity surfaces: tasks/problem matchers, MCP servers, agent sessions, and subagents
- For Tesseract, these classes should not be treated as interchangeable. Some are behavior shapers, some are explicit operator-action surfaces, and some are full orchestration hosts.
- The main architectural risk is not missing capability. It is accidentally creating a second hidden workflow model outside the Tesseract artifact envelope.

## Synthesis

### What belongs in the first scaffold

- Repo-local behavior shaping:
  - `AGENTS.md`
  - `.github/copilot-instructions.md`
  - path-specific `.instructions.md`
  - a small number of prompt files
- Explicit operator actions:
  - VS Code tasks over existing `tesseract` commands
  - problem matchers over inbox, hotspot, and approval/rerun outputs

### What belongs in the second scaffold

- Agent Skills for portable, richer workflows once the team wants the same flows in Copilot CLI or coding-agent contexts.
- Custom agents for planner, reviewer, or dogfood coordinator roles once the base action surfaces are stable.

### What should stay optional until later

- Chat Participant API, because it would create a bespoke conversational control plane.
- Language Model Tool API, unless Tesseract decides to ship a first-party extension with explicit tool wrappers.
- MCP, unless Tesseract wants portable read-only artifact resources or cross-environment tools.

## Important Uncertainties

- Agent Skills support is the clearest documentation mismatch. VS Code docs present it as current product surface; GitHub docs still describe stable VS Code support as pending. Treat that as a real rollout unknown, not as a wording issue.
- Prompt files and custom agents are heavily documented, but actual enterprise policy, agent enablement, and model availability can still constrain what a team sees in the VS Code UI.
- The ecosystem is rich enough that Tesseract could implement the same capability in multiple ways. That makes the repository-side contract choice more important, not less.
